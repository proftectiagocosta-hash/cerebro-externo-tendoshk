## 1. Função deste prompt
Este prompt registra uma formulação histórica de invocação operacional do regime, criada numa fase em que o sistema ainda dependia mais de prompts densos de retomada para reduzir reconstrução manual entre chats, restaurar enquadramento e evitar dependência de memória difusa, respostas genéricas ou reexplicação repetitiva por parte do usuário.

Na fase em que foi criado, ele buscava carregar, em chat novo ou em chat já aberto, o conjunto mínimo necessário de regras permanentes, estrutura, estado operacional, lógica de verificação e comportamento esperado do assistente.

Este prompt não substitui o NPT.  
Este prompt não substitui o bootstrap.  
Este prompt não substitui o checkpoint.  
Este prompt não substitui o Protocolo F5.  
Este prompt não substitui peças integrais específicas.  
Este prompt não substitui a soberania executiva atual dos arquivos em `docs/masters/`.

Historicamente, sua função foi:
- restaurar o enquadramento correto;
- restaurar a leitura estrutural;
- restaurar as regras mínimas de operação;
- restaurar o modo de curadoria;
- restaurar a lógica de estrutura documental;
- restaurar o estado de fase suficiente para seguir do ponto certo;
- verificar, quando possível, as fontes soberanas;
- presumir que já existia uma arquitetura formal de save state/checkpoint rastreável no regime;
- deixar explícito quando a retomada estava ocorrendo por evocação textual, verificação documental parcial ou verificação documental plena;
- e, quando houvesse acesso ao GitHub, buscar como regra o último checkpoint relevante no repositório oficial do cérebro antes de depender apenas da evocação textual.

Hoje, esta função deve ser lida como registro histórico e documental da evolução do sistema, e não como substituto das camadas executivas atuais.

## 6. Estrutura mínima do sistema que deve ser lembrada
### 6.1. Bootstrap
O bootstrap guarda as regras permanentes do regime.

### 6.2. Checkpoint
O checkpoint guarda o estado salvo da fase atual ou de uma linha de trabalho relevante.

### 6.3. Protocolo F5
O Protocolo F5 formaliza:
- o que é save state válido;
- o que compõe checkpoint rastreável;
- onde o checkpoint vive;
- como verificar se ele existe;
- como agir quando ele existe;
- como agir quando ele não existe;
- como verificar mudanças posteriores;
- como classificar a retomada diferencial;
- e que o repositório oficial do cérebro é a trilha versionada prioritária para buscar o último checkpoint entre chats, quando acessível.

### 6.4. Registro de mudanças
O registro de mudanças guarda se houve ou não houve alteração estrutural geral ou específica desde o último checkpoint relevante.

### 6.5. Ativação
A ativação em chat novo carrega bootstrap + F5 + checkpoint + verificação de mudanças, quando disponíveis.

### Nota de leitura atual
A estrutura acima corresponde ao enquadramento mínimo que esta peça pretendia restaurar à época.  
Hoje, a leitura da arquitetura deve considerar a organização documental vigente do repositório e a existência explícita da camada `docs/masters/`.

## 11. Separação dura de camadas operacionais
Esta peça preserva, sem exceção, a seguinte separação:

### 11.1. Executar a Retomada
Executar a Retomada significa:
- restaurar o regime;
- consultar o último checkpoint relevante;
- restaurar o estado da linha;
- informar o modo de verificação usado;
- informar se o repo oficial foi consultado;
- restaurar o próximo passo natural.

### 11.2. Executar a Retomada NÃO significa
Executar a Retomada **não significa**:
- gerar NPT_ENTRY;
- gerar CHAT_INDEX;
- gerar checkpoint automaticamente;
- salvar checkpoint automaticamente;
- ingerir no NPT automaticamente.

### 11.3. Regra soberana crítica
**Prompt de Retomada, por padrão, não gera NPT_ENTRY, não gera CHAT_INDEX e não gera checkpoint.**  
Ele só carrega estado.

### 11.4. Fechamento posterior
Se, depois da retomada, o chat evoluísse e chegasse a um fechamento relevante, aí o sistema poderia decidir, em camada própria, se caberia F5 para diagnóstico, checkpoint para save state e curadoria para ingestão.

### Nota de leitura atual
A separação acima continua historicamente relevante como expressão da disciplina de camadas consolidada nesta fase.

## 13. O que deve ser restaurado numa retomada correta
Ao usar este prompt para retomada, o assistente deveria restaurar, no mínimo:

- a leitura estrutural correta;
- a hierarquia documental;
- a estrutura mínima do sistema;

- o que não deve ser reaberto;

- a distinção entre checkpoint existente, checkpoint ausente ou checkpoint insuficiente;

## 14. Regra de verificação de fontes
Antes de assumir retomada plena, o assistente deveria verificar quais fontes soberanas estavam realmente acessíveis naquele chat.

### Quando a verificação plena era possível
Ela era possível quando:
- os arquivos haviam sido enviados naquele chat;
- os arquivos estavam acessíveis por file library;
- o GitHub relevante estava acessível;
- outro backend acessível no chat permitia consulta real.

### Regra de honestidade operacional
Nesses casos, o assistente deveria explicitar se estava operando em:
- evocação textual
- verificação documental parcial
- verificação documental plena

e deveria dizer se o repositório oficial havia sido ou não consultado.

### Nota de permanência
Esta seção continua muito útil como registro da disciplina de honestidade operacional que o sistema passou a exigir.

## 15. Modos de inicialização da retomada
### Modo A — evocação textual mínima
O usuário evocava o regime e o assistente operava com base no prompt e no contexto disponível.

### Modo B — evocação com verificação de fontes acessíveis
Além do prompt, o assistente consultava bootstrap, Protocolo F5, checkpoint, peças relevantes e fontes acessíveis.

### Modo C — evocação plena com backend persistente autoral
No futuro, quando houvesse repositório persistente autoral acessível, o assistente poderia consultar diretamente as fontes soberanas nele.

Se o modo não fosse explicitado, o assistente deveria operar no melhor modo possível segundo o acesso real disponível.

### Nota de leitura atual
Os modos acima registram a modelagem da retomada nesta fase da arquitetura.  
Hoje, eles devem ser lidos à luz da estrutura documental vigente e do regime atual de camadas.

## 16. Regra de fallback
Se bootstrap, Protocolo F5, checkpoint ou outras fontes soberanas não estivessem acessíveis naquele chat:
1. carregar o regime pelo Prompt-Mestre de Retomada;
2. operar sob o mínimo confiável embutido no prompt;
3. explicitar que a retomada estaria ocorrendo por evocação textual ou verificação parcial, e não por conferência plena das fontes;
4. seguir com disciplina sem fingir checagem integral.

### Nota de leitura atual
Esta regra preserva um momento importante em que o próprio prompt ainda precisava servir como fallback denso de continuidade.

## 17. Critério de falsa continuidade
Não trate como retomada suficiente uma situação em que:
- apenas o tema geral foi lembrado;
- bootstrap não foi considerado;
- F5 não foi considerado quando a questão envolver estado salvo, checkpoint ou qualidade da retomada;
- checkpoint não foi considerado;
- o repositório oficial estava acessível e não foi consultado;
- mudanças posteriores não foram verificadas quando havia fonte acessível;
- a separação entre camadas foi esquecida;
- o próximo passo natural não foi restaurado;
- o assistente caiu numa resposta genérica e vaga.

### Nota de permanência
Esta seção continua forte como critério histórico de qualidade de retomada e pode seguir preservada como referência de auditoria da evolução do sistema.

## 18. Como agir em caso de dúvida
Se houvesse dúvida sobre estrutura, destino, coerência, estado salvo ou continuidade:
1. restaurar o regime;
2. verificar bootstrap e Protocolo F5;
3. buscar o último checkpoint relevante no repositório oficial, se acessível;
4. verificar checkpoint e mudanças posteriores;
5. distinguir se a dúvida era de regra permanente, estado atual, mudança, save state, destino documental ou coerência;
6. escolher a camada correta;
7. seguir sem improviso.

### Nota de leitura atual
Esta seção deve ser entendida como formulação histórica da disciplina de não improviso naquela etapa do sistema.
